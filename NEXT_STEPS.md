## 1. ТРЕК A — Eval & observability (фундамент, делаем первым) 🎓

> **Выбранный технический трек (приоритет):** A (Eval) → C (Advanced retrieval, §3) → B (Agentic RAG, §2). Сначала строим измеримость, затем под метрики гоняем SOTA-ретрив, затем агентность. Каждый шаг — мини-модуль с числом «до/после».

Сквозная мысль всех модулей: **настраивать на глаз нельзя, нужно мерить**. Сейчас у нас нет ни одной метрики качества поиска — это блокер для всего продвинутого ретрива.

- 🔴🎓 **Eval-харнесс для ретрива.** Набор из 15–30 пар «вопрос → какой чанк/документ правильный». Скрипт считает **Recall@k** (попал ли нужный чанк в кандидатов) и **MRR/nDCG** (на каком месте). Тогда любые изменения (размер чанка, `rerank_candidates`, dense-only vs hybrid vs +rerank, HyDE, late-interaction) сравниваются числом, а не ощущением.
  - Это разблокирует осознанную настройку `chunk_size` (рычаг из Модуля 1) и `rerank_candidates` (Модуль 3), и весь §3.
- 🟡🎓 **RAGAS — метрики качества всего RAG, не только ретрива.** `faithfulness` (ответ не галлюцинирует за пределы контекста), `answer_relevancy`, `context_precision/recall`. LLM-as-judge поверх golden-набора. Закрывает дыру «ретрив нашёл, но ответ всё равно мимо».
- 🟡🎓 **LangSmith — трейсинг графов.** Подключить через `LANGCHAIN_TRACING_V2` — видеть каждый узел LangGraph, промпты, токены, латентность, кандидатов ретрива. Бесценно для отладки агентов из §2 (что LLM реально решила, какие запросы послала).
- 🟢 **Логировать запрос → переписанный запрос → кандидаты → финальный top-k.** Хотя бы в `logger.debug` (дешёвая альтернатива LangSmith на старте).

## 2. ТРЕК B — Agentic RAG (вершина, делаем после A+C) 🎓

Сейчас mentor — **фиксированный граф**: всегда ищет ровно один раз. Цель трека — превратить его в агента, который сам управляет ретривом и проверяет себя. По возрастанию сложности:

- 🟡🎓 **Tool-calling mentor.** Дать LLM `retrieve` как **инструмент**, чтобы она сама решала: нужен ли поиск вообще (на «привет» — нет); какой запрос послать; сходить ли в базу **несколько раз** (multi-hop); когда контекста достаточно. Паттерн `create_react_agent` или ручной цикл `agent ⇄ tools` с условным ребром. Точка входа — [mentor_agent.py](backend/app/services/agents/mentor_agent.py).
- 🟡🎓 **Query decomposition.** Сложный вопрос («сравни X и Y по критерию Z») LLM разбивает на под-запросы, ищет по каждому, агрегирует. Узел перед retrieve.
- 🔴🎓 **CRAG (Corrective RAG).** После ретрива лёгкий грейдер оценивает релевантность кандидатов; если плохо — переписать запрос и переискать (или fallback). Условные рёбра в LangGraph.
- 🔴🎓 **Self-RAG / reflection-петля.** Агент критикует собственный черновик ответа («подкреплён ли контекстом? полон ли?») и при необходимости доискивает/переписывает. Здесь особенно нужен LangSmith (§1), чтобы видеть петли.

Меряем эффект через RAGAS (§1): `faithfulness` и `answer_relevancy` до/после.

## 3. ТРЕК C — Advanced retrieval (делаем после A, каждый шаг меряем) 🎓

Раньше было «по желанию» — теперь это ядро технического трека. Каждый пункт — отдельный эксперимент: меняем один метод, гоняем eval-харнес (§1), фиксируем Recall@k/nDCG «до/после». Без §1 этот раздел делать бессмысленно.

- 🔴🎓 **Late-interaction (ColBERT / ColPali).** SOTA-замена single-vector эмбеддингам: токен-уровневые мульти-векторы + MaxSim вместо одного вектора на чанк. Qdrant нативно поддерживает multivector. Самый «высокотехнологичный» рычаг качества, но дорогой по памяти/индексу → обязательно мерить, что окупается.
- ✅🎓 **HyDE (Hypothetical Document Embeddings).** _(Модуль 7, эксп.2 — СДЕЛАНО, измерено.)_ LLM генерит гипотетический ответ, ищем по его эмбеддингу (dense-сторона; sparse на исходном запросе). **Вывод: на нашем корпусе не помогает** — dense уже у потолка после Модуля 6, гипо-док вносит дрейф (dense+hyde строго хуже, hybrid+hyde теряет Hit@1/MRR, hybrid+hyde+rerank ≡ hybrid+rerank). `hyde_enabled=False`. Код за флагом в `hyde.py`. Бонус-инсайт: сильный reranker нивелирует выбор ретривера 1-й стадии. См. LEARNING_NOTES Модуль 7.
- ✅🎓 **Умный чанкинг — рекурсивный сплиттер.** _(Модуль 8, эксп.3 — СДЕЛАНО, измерено.)_ Заменил скользящее окно по token-id на рекурсивный сплиттер по естественным границам (абзац→строка→предложение, token-фолбэк только если предложение само длиннее лимита) + overlap целыми предложениями. **Вывод: на нашем корпусе ретрив-метрику не двигает** — Δ≈0 по всем стратегиям (@1/@3/@5 заморожены, шум на 4-м знаке только @10). Причина: гранулярность та же (81 чанк), метрики у потолка, голд-термины сидят внутри чанков, а не на релоцируемых границах. Но **оставил включённым** (`chunking_strategy="recursive"`): нейтрален к ретриву, бесплатен, чанки на границах предложений → чище контекст для LLM (разорванный ACID склеился). За флагом `token` — легаси-окно для воспроизведения baseline. См. LEARNING_NOTES Модуль 8.
- 🟡🎓 **Умный чанкинг — semantic / late chunking ← следующий кандидат.** Осталось: **semantic chunking** (граница там, где «прыгает» смысл — требует эмбеддить каждое предложение), **late chunking** (эмбеддить весь документ, потом резать — сохраняет контекст). Тяжелее по памяти (Docker ~2 ГБ), отдельный эксперимент. Точка входа — [chunker.py](backend/app/services/ingestion/chunker.py).
- 🟡🎓 **Parent-document retrieval.** Искать по мелким чанкам (точность), а в LLM отдавать родительский крупный блок / соседние чанки (контекст). Закрывает «нашли нужное предложение, но без окружения».
- 🟡 **Фильтрация по метаданным.** Qdrant фильтрует по payload (`document_id`, `file_type`) до поиска — основа для «искать только в этих документах» (нужно §6-A). Инфраструктура `_build` уже в [qdrant_store.py](backend/app/services/rag/qdrant_store.py).
- 🟢 **Дедуп/MMR.** При выдаче top-k убирать почти одинаковые чанки (maximal marginal relevance) — разнообразие контекста.
- 🟡 **Лучше эмбеддинги.** `all-MiniLM-L6-v2` (384, лимит 256) — компромисс. Можно `bge-base`/`e5`/`gte` (больше контекст и качество), но смена размерности → пересоздание коллекции + re-ingest.

## 6. Интерактивное обучение (Phase 6) — прикладная поверхность

> **Роль в роадмапе:** это не технический трек, а **витрина**, на которой проверяются техники из §1–3. Приоритет ниже A/B/C. Полезные связки: §6-A (фильтр по коллекции) — прямой потребитель metadata-фильтра из §3; квиз/чат по теме — хороший прикладной кейс для замера Agentic RAG (§2). Делать выборочно, когда нужна демонстрация техник «в продукте».

Превращает разрозненные загрузки в **учебные единицы**: темы из нескольких документов, квизы по отдельным разделам, упражнения с реальной проверкой. Опирается на уже существующие поля (`h1/h2/h3_title` в чанках, `source_chunk_id` в вопросах, payload-фильтры Qdrant) — это в основном новые ручки + миграция, не переписывание.

> **Зависимости от других разделов:**
> - A и B меняют ретрив (фильтр по документам, квиз по разделу) → их качество имеет смысл оценивать **после eval-харнеса (§1)**, иначе настраиваем «на глаз».
> - C («объясни своими словами» + Сократ) переиспользует **tool-calling ментора (§2)** — делать после него.
>
> **Терминология (важно):** в коде уже есть `PlanTopic` (узел дерева плана) и квиз умеет генериться по `topic_id` плановой темы. Чтобы не плодить коллизию, новую сущность-группу документов называем **`Collection`** на уровне БД/кода, а в UI показываем как «тему». Раздел документа идентифицируем парой **`(document_id, heading_path)`**, а не голой строкой заголовка (заголовки не уникальны между документами).

### Ограничения, которые расшиваем

- Квиз привязан к **целому документу или теме плана** ([quizzes.py:74-93](backend/app/api/v1/quizzes.py#L74-L93)) — берёт первые 10 чанков. Нельзя сделать квиз «по разделу H2».
- Нет связи **«Тема = несколько документов»**: `PlanTopic` не ссылается на `Document`/чанки.
- `short_answer` проверяется строковым `==` ([quizzes.py:170](backend/app/api/v1/quizzes.py#L170)) — почти всегда ложно-отрицательно.
- Прогресс не связан с покрытием материала (нельзя сказать «пройдено 60% разделов»).

### A. Темы (`Collection`) как объединение документов 🔴🎯

- 🟡 **Модель `Collection` + связь many-to-many с `documents`.** Новые таблицы `collections` и `collection_documents`. Миграция Alembic (по образцу `0001_initial.py`). Имя `Collection` — чтобы не путать с `PlanTopic`; в UI зовём «темой».
- 🟡 **Фильтр поиска по коллекции.** В `hybrid_search` добавить `document_ids: list[UUID] | None` → Qdrant `Filter(must=[FieldCondition(key="document_id", match=MatchAny(...))])`. Точка входа — [retriever.py](backend/app/services/rag/retriever.py) + [qdrant_store.py](backend/app/services/rag/qdrant_store.py). _(Оценить эффект после eval-харнеса §1.)_
- 🟢 **Сфокусированный чат.** Параметр `collection_id` в создании сессии → ментор отвечает только по документам коллекции (переиспользует фильтр выше).
- 🟡 **UI: CRUD тем.** Страница `/collections` (в навигации — «Темы») — создать, прикрепить документы галочками, кнопки «План» / «Квиз» / «Спросить ментора».

### B. Квизы и материал по разделам 🟡🎯

- 🟢 **Outline-вью документа.** Строим оглавление из `h1/h2/h3_title` существующих чанков (данные уже есть, новых полей не нужно). Эндпоинт `GET /knowledge/documents/{id}/outline`.
- 🟡 **Квиз по разделу.** `GenerateQuizRequest` принимает `(document_id, heading_path)` → фильтруем чанки по этой паре (а не по голой строке заголовка — он не уникален между документами) → 3-5 точечных вопросов. Расширяет [quizzes.py:74](backend/app/api/v1/quizzes.py#L74).

### C. Интерактивные упражнения 🔴🎯

- 🟡🎓 **LLM-грейдинг open-ended.** Для `short_answer` оценку даёт LLM (верно / частично / неверно + объяснение), а не `==`. Чинит реальный баг в [quizzes.py:170](backend/app/api/v1/quizzes.py#L170). ### D. Адаптивность и прогресс 🟡

- 🟡 **Карта покрытия.** Привязать попытки квизов к разделу/теме → «10 из 14 разделов отработаны, средний балл 78%». Расширяет `/progress/summary`.

## Рекомендуемый порядок (технический трек)

Выбранный фокус — высокотехнологичные RAG/agent-инструменты, измеряемые числом. Хребет:

1. **Хвосты (§0)** — система честно зелёная, включая UI и квизы.
2. **ТРЕК A — Eval & observability (§1):** eval-харнес (Recall@k/MRR/nDCG) → RAGAS (faithfulness/relevancy) → LangSmith-трейсинг. Фундамент: без чисел остальное настраивается «на глаз».
3. **ТРЕК C — Advanced retrieval (§3):** по одному эксперименту, каждый меряем через A. Порядок по соотношению эффект/цена: ~~смена эмбеддингов (Модуль 6, exp.1 ✅)~~ → ~~HyDE (Модуль 7, exp.2 ✅ — не помог, за флагом)~~ → ~~рекурсивный чанкинг (Модуль 8, exp.3 ✅ — Δ≈0, оставлен включённым за качество)~~ → **semantic/late chunking ← следующий** → parent-document → MMR → late-interaction (ColBERT).
4. **ТРЕК B — Agentic RAG (§2):** tool-calling → query decomposition → CRAG → Self-RAG/reflection. Меряем RAGAS-ом, отлаживаем LangSmith-ом.
5. **Прикладная витрина (§6):** выборочно — когда хочется показать техники «в продукте» (фильтр по коллекции поверх metadata-фильтра §3, чат/квиз по теме как кейс для агента).

Базовая функциональность (§5: auth, импорт из GitHub) и надёжность (§4) — по необходимости, вне основного учебного фокуса.
