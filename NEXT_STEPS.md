## 1. Главный RAG-шаг: измеримость («перестать гадать»)

Сквозная мысль всех модулей: **настраивать на глаз нельзя, нужно мерить**. Сейчас у нас нет ни одной метрики качества поиска.

- 🔴🎓 **Eval-харнесс для ретрива.** Набор из 15–30 пар «вопрос → какой чанк/документ правильный». Скрипт считает **Recall@k** (попал ли нужный чанк в кандидатов) и **MRR/nDCG** (на каком месте). Тогда любые изменения (размер чанка, `rerank_candidates`, dense-only vs hybrid vs +rerank) сравниваются числом, а не ощущением.
  - Это разблокирует осознанную настройку `chunk_size` (рычаг из Модуля 1) и `rerank_candidates` (Модуль 3).
- 🟡 **Логировать запрос → переписанный запрос → кандидаты → финальный top-k.** Хотя бы в `logger.debug`, чтобы видеть, что реально уходит в LLM.

## 2. Вершина агентов: tool-calling mentor 🎓

Сейчас mentor — **фиксированный граф**: всегда ищет ровно один раз. Следующий уровень — дать LLM `retrieve` как **инструмент**, чтобы она сама решала:
- нужен ли поиск вообще (на «привет» / «спасибо» — нет);
- какой запрос послать (и переписать его);
- сходить ли в базу **несколько раз** (multi-hop для составных вопросов);
- когда контекста достаточно, чтобы отвечать.

В LangGraph это паттерн `create_react_agent` или ручной цикл `agent ⇄ tools` с условным ребром. Точка входа — [mentor_agent.py](backend/app/services/agents/mentor_agent.py). 🟡🎓

## 3. Дальнейшие улучшения RAG (по желанию, каждый — мини-модуль)

- 🟡🎓 **Умный чанкинг.** Сейчас режем по токенам внутри секции. Варианты лучше: рекурсивный сплиттер по границам предложений/абзацев (не рвать середину фразы), late chunking, semantic chunking. Точка входа — [chunker.py](backend/app/services/ingestion/chunker.py).
- 🟡 **Фильтрация по метаданным.** Qdrant умеет фильтровать по payload (`document_id`, `file_type`) до поиска — например «искать только в этом документе». Уже есть `_build` инфраструктура в [qdrant_store.py](backend/app/services/rag/qdrant_store.py).
- 🟡 **Контекст для цитат / окно вокруг чанка.** Возвращать соседние чанки того же документа, чтобы LLM видела больше контекста (parent-document retrieval).
- 🟢 **Дедуп/MMR.** При выдаче top-k убирать почти одинаковые чанки (maximal marginal relevance) — разнообразие контекста.
- 🟡 **Лучше эмбеддинги.** `all-MiniLM-L6-v2` (384, лимит 256) — компромисс. Можно взять модель с бóльшим контекстом/качеством (`bge-base`, `e5`), но это смена размерности → пересоздание коллекции + re-ingest.

## 4. Надёжность и продакшн

- 🟡 **Покрыть тестами агенты и API.** Сейчас тесты только на chunker/sparse/reranker. Добавить: `contextualize_query`, парсинг цитат, planner structured output, smoke-тесты роутов (httpx + ASGITransport).
- 🟡 **Обработка ошибок ингестии.** Что если PDF битый, документ пустой, эмбеддинг упал? Статус `error` есть — проверить, что он реально проставляется и виден в UI.
