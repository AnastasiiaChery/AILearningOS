## Модуль 1 — Чанкинг и эмбеддинги

### Концепция
- Эмбеддинг-модель превращает текст в **один вектор фикс. длины** (384). Один вектор на весь документ «усредняет» смысл → деталь тонет. Поэтому документ режут на **чанки**, каждый → свой вектор, поиск идёт по чанкам.
- **Жёсткий факт:** у каждой модели есть лимит входа в токенах. У `all-MiniLM-L6-v2` это **256 токенов**. Всё длиннее — **молча обрезается**, остаток текста никогда не попадает в вектор.
- `normalize_embeddings=True` → векторы единичной длины, тогда **cosine == dot product**, что и ожидает Qdrant с `Distance.COSINE`.

### Проблема в коде (было)
- `_estimate_tokens = len(text)//4` — «токены» считались по символам, а не реальным токенайзером.
- `_split_by_size`: окно бралось как `max_tokens*4` **слов** (`512*4=2048` слов), хотя комментарий про «4 символа на слово». Токены/слова/символы перемешаны.
- `chunk_size=512` в конфиге — **вдвое больше лимита модели (256)**.
- Баг **латентный**: на мелких документах секции маленькие, до `_split_by_size` дело не доходит — всё «выглядит нормально».

### Моё рассуждение (разбор на примере секции 1500 слов)
1. В один чанк попадёт **~2048 слов** (по формуле `words[start : start + 512*4]`).
2. Это **~2662 реальных токена** (1 слово ≈ 1.3 токена).
3. → Модель **обрежет** до первых 256 токенов.

**Поправка (ключевой "ага"):** 256 из 2662 — это **не «чуть больше половины», а < 10%**. То есть в вектор попадал примерно первый абзац, а **~90% секции выбрасывалось** и было невидимо для поиска. Чанк при этом лежит в БД «целиком» — поэтому баг и коварен.

### Моя гипотеза «нужно делать чанки меньше» — и важная развилка
Смешались **две разные вещи**:
1. **Потеря из-за обрезки** лечится **не** уменьшением чанка, а тем, что мы режем секцию на **много** чанков и покрываем её **целиком** (+ overlap, чтобы мысль на стыке не рвалась). Информация больше не теряется в принципе.
2. **Размер чанка** — отдельный **рычаг точности**, классический trade-off:

| Чанки меньше (128–180) | Чанки больше (≈256) |
|---|---|
| острее семантика, меньше шума | больше контекста в одном куске |
| идея может разорваться; больше точек в БД | вектор «усредняет» смысл, факт может раствориться |

Вывод: «меньше чанк» **не возвращает** потерянную инфу (её возвращает разбиение), а повышает точность ценой контекста. Правильный инженерный путь — **разумный дефолт (256) + замер** (настроим эмпирически в Модуле 3 с reranker’ом), а не подбор «на глаз».

### Решение (стало)
- Счёт и нарезка — **родным токенайзером модели** (`get_embedder().tokenizer`): лимит 256 определён именно в её WordPiece-токенах, поэтому это единственный честный источник истины (tiktoken от OpenAI — другой словарь, дал бы расхождение).
- `_split_by_tokens`: sliding-window по реальным id с overlap по токенам, decode обратно в текст.
- `_effective_max_tokens = min(chunk_size, model_max) - 16`. **Headroom 16** — на спец-токены `[CLS]/[SEP]` **и** на дрейф round-trip: окно из N id, декодированное в текст и заново закодированное при эмбеддинге, может вернуться как на пару токенов больше (`decode∘encode` у WordPiece **не тождественен**).
- `embedder.py`: добавлены `aembed_texts/aembed_query` через `asyncio.to_thread` — `model.encode` это блокирующая CPU-операция, прямой вызов в async **заморозил бы event loop** на всё время кодирования.

### Проверка на данных
- Секция **1440 слов**: было — 1 обрезанный чанк (теряем ~90%); стало — **9 чанков по ≤240 токенов**, покрытие полное.
- `tests/test_chunker.py` — 4 регрессии (≤ лимит, многокусочное разбиение, headroom, покрытие первого/последнего токена) — **зелёные**.

## Модуль 2 — Sparse / BM25 и гибридный поиск *(в процессе)*

### Концепция
- **Dense вектор** — семантика (384 числа). Ловит перефразирование (*«поиск по смыслу»* ≈ *«semantic retrieval»*). Слепое пятно — **точные редкие термины** (имена функций, артикулы, аббревиатуры вроде `RRF`).
- **Sparse вектор** — лексика. Размер во весь словарь (десятки тысяч), почти всё нули; ненулевые = веса реально встретившихся слов. Ловит **точное совпадение**.
- **BM25** — классические веса sparse:
  - **TF** — чем чаще слово в документе, тем выше вес, но с **насыщением** (5-е упоминание < 2-го).
  - **IDF** — чем в **меньшем числе документов корпуса** слово встречается, тем оно ценнее (`the` ≈ 0; `qdrant` — большой). **IDF считается по всему корпусу, не внутри одного документа.**
- **Гибрид + RRF** (Reciprocal Rank Fusion): объединяем не сырые скоры (косинус и BM25 в разных шкалах), а **ранги**: `score = Σ 1/(k + rank)`. Устойчиво к разным шкалам.

### Проблема в коде (`qdrant_store.py:30-47`, тот же код при загрузке и при запросе)
```python
idx = hash(token) % 65536            # слово → колонка вектора
tf  = count / total
idf = math.log(1 + 1/(count + 1))    # «IDF» по частоте в ОДНОМ тексте
```
Две гипотезы дефекта (проверим экспериментом):
- **А (hash-соль):** `hash()` для строк в Python солится случайно при каждом запуске процесса (`PYTHONHASHSEED`). Если бэкенд перезапущен между загрузкой и запросом — `hash("qdrant") % 65536` даст **другую колонку** → слова перестают совпадать → keyword-поиск тихо деградирует.
- **Б (ложный IDF):** формула использует частоту в одном тексте, а не долю документов корпуса → редким и частым словам даёт почти одинаковый вес → смысл IDF теряется.

### Моё рассуждение (и эксперимент)
- **Часть А** — гипотеза «hash совпадёт» **опровергнута экспериментом**: `hash("qdrant") % 65536` дал 1909 / 38650 / 44106 (хост) и 6896 / 10086 (контейнер) — каждый запуск процесса разный. `PYTHONHASHSEED` не задан → рандомизация включена (дефолт CPython с 3.3). Вывод: после рестарта слово ложится в **другую колонку** → перестаёт находить себя. Тихая деградация (dense ещё что-то находит — кажется, что «работает»).
- **Часть Б** — «формула не даёт одинаковый вес»: **наполовину верно**. Формула не константа (падает с `count`: 0.4055 → 0.0099), но зависит от **неправильной переменной** — частоты **внутри одного текста**. Эксперимент: `the`(1 раз)=0.4055 и `qdrant`(1 раз)=0.4055 — **идентично**; настоящий корпусный BM25-IDF: `the`=0.05, `qdrant`=5.66 (×100). Корень: IDF требует статистики корпуса, а функция видит один текст за раз.

### Решение (свой TF + стабильный хеш + Qdrant IDF) — выбрано осознанно
1. **Стабильный токен→id**: `blake2b(token)` в пространство `2^20` вместо солёного `hash()` → детерминирован между процессами (чинит дефект №1).
2. **В вектор кладём только TF** c BM25-насыщением `count*(k1+1)/(count+k1)`, `k1=1.5`, **без** length-norm (он требует корпусного avgdl) и **без** самодельного idf.
3. **IDF делегирован Qdrant** через `SparseVectorParams(modifier=Modifier.IDF)` в `lifespan.py` — сервер считает IDF по реальной document-frequency корпуса при запросе (чинит дефект №2, переживает рестарты).
4. **Агрегация по индексу** (коллизии хеша) — Qdrant требует уникальные индексы в sparse-векторе.
5. Стоп-слова **не нужны списком** — корпусный IDF сам придавливает их к нулю.
6. ⚠️ Коллекцию пришлось **пересоздать**: модификатор задаётся при создании; existing-коллекция без него не обновится → delete + restart (lifespan создаёт заново) + re-ingest.

### Проверка на данных
- **Детерминизм/рестарт:** `token_id('rrf')=462471`, `token_id('qdrant')=277249` — **идентичны до и после рестарта**; sparse-only поиск «RRF…» → чанк «Hybrid Search», score **7.4133**, тот же после рестарта.
- **IDF-эффект (корпусный):** sparse-score `qdrant`/`hybrid`=1.674 (редкие), `the`=0.297 (стоп-слово придавлено автоматически). Нюанс: IDF относителен к корпусу; на 7 чанках статистика шумная (`is`=1.18, т.к. реже `the`).
- **Полный гибрид:** `hybrid_search` → 5 чанков, dense+sparse→RRF, top = «Hybrid Search» (rrf_score 1.0).
- `tests/test_sparse.py` (детерминизм, уникальные индексы, насыщение TF, «нет idf в значениях») + `test_chunker.py` — **8/8 зелёные**.

### Открытый хвост (не блокер)
- `qdrant-client` в образе — **1.18.0**, сервер — **1.12.4**: предупреждение о несовпадении minor-версий (>1). Сейчас всё работает; чисто было бы выровнять (поднять образ Qdrant до 1.18.x или закрепить клиент ~1.12). Решим при желании.

## Модуль 3 — Reranking

### Концепция
- **Bi-encoder** (наш `all-MiniLM-L6-v2`): кодирует запрос и документ **раздельно** в векторы, сравнивает косинусом. Вектор документа можно **предпосчитать** при загрузке → быстрый поиск. Цена — грубость: в момент кодирования документ «не знает» о запросе, нюанс усреднён.
- **Cross-encoder** (reranker `cross-encoder/ms-marco-MiniLM-L-6-v2`): берёт пару **(запрос, документ) вместе**, cross-attention позволяет токенам запроса смотреть на токены документа → точная релевантность одним числом. Цена — **нечего предпосчитать**, нужен forward-pass на каждую пару при запросе.
- **Двухстадийная схема:** Стадия 1 (гибрид, дёшево) → top-N кандидатов (recall). Стадия 2 (cross-encoder, дорого) → пересчёт N пар, top-k (precision).

### Моё рассуждение
- **А:** cross-encoder видит обе стороны одновременно → анализирует **взаимодействие** токенов (cross-attention), а не расстояние между двумя готовыми векторами. Bi-encoder сжимает каждую сторону до их «встречи» — нюанс уже потерян.
- **Б:** дорого, т.к. на каждый запрос нужен прогон трансформера для **каждой** пары; ничего не предпосчитать. Узкое место при 100k чанков: reranker = **O(N) прогонов/запрос** (секунды-минуты), а bi-encoder+ANN = **~O(1) кодирование + O(log N) поиск** (вектора посчитаны 1 раз при загрузке). Поэтому reranker — только по маленькому пулу кандидатов.

### Решение
- Новый сервис `services/rag/reranker.py`: `CrossEncoder` синглтон (`lru_cache`), `rerank_sync` (scores → sort → top_k), `arerank` через `asyncio.to_thread` (CPU-bound, как эмбеддер).
- `retriever.hybrid_search(query, top_k, rerank)`: стадия 1 тянет `rerank_candidates=20` (шире для recall), стадия 2 — cross-encoder → `retrieval_top_k=5`. Флаг `rerank` позволяет сравнивать с чистым RRF.
- Конфиг: `rerank_enabled`, `rerank_candidates`, `reranker_model`. Reranker греется на старте в `lifespan.py`.
- Зависимостей не добавляли — `CrossEncoder` уже в `sentence-transformers` (модель качается с HF в рантайме, как эмбеддер).

### Проверка на данных
Запрос *«why does a cross-encoder improve the order of search results»* по обогащённому корпусу:
- **RRF only (top-5):** скоры плоские (1.0, 0.58, 0.50, 0.42, 0.33).
- **+ rerank (top-5):** единственный релевантный кусок «Why Reranking Helps» = **+5.82**, остальные **−8…−11** — резкое разделение. Состав top-5 переставлен (вошла Quantization из пула 20, выпала Cosine).
- Вывод: cross-encoder даёт **дискриминативный** сигнал (что реально отвечает на вопрос), RRF — лишь относительный ранг.
- `tests/test_reranker.py` (ранжирование по релевантности + усечение + пустой вход) — часть набора **10/10 зелёных**.

## Модуль 4 — Дизайн агентов LangGraph

### Концепция
- **Цепочка/фиксированный граф** — путь зашит. Наш mentor: `retrieve → generate`, всегда ищет один раз сырым последним сообщением.
- **Агент** — LLM сам решает (через tool calls): искать или нет, какой запрос, сходить ли ещё раз (multi-hop), когда остановиться.

### Проблема (показано на данных)
Follow-up-запросы достают мусор, т.к. retrieve берёт сырой `user_message`:
- `What is RRF…` → топ ce=+8.65 (ок); `can you give an example of that?` → ce=−3.19; `tell me more about it` → ce=−9.34 (всё нерелевантно — слова `it/that` без темы). Контекст диалога терялся на этапе поиска.

### Моё рассуждение
1. Нужную тему хранит **история диалога** (`state["history"]`).
2. Превращение «that» → «Reciprocal Rank Fusion» — работа **LLM**, не кода; шаг встаёт **новым узлом перед retrieve**.

### Решения (3 фикса)
**№6 — контекстуализация запроса.** Новый узел `contextualize_query`: `history + user_message → standalone search query` через LLM (только при наличии истории — первое сообщение самодостаточно, экономим вызов). Граф: `contextualize → retrieve → generate`. Поле `search_query` в `MentorState`. Проверка: `can you give an example of that?` → **«example of Reciprocal Rank Fusion in hybrid search»**; `tell me more about it` → **«What is Reciprocal Rank Fusion in hybrid search»**.

**№7 — настоящий стриминг.** Было: `ainvoke` целиком → разбивка по словам (пользователь ждёт весь ответ). Стало: узел `generate` использует `llm.astream` (`streaming=True`), а `chat.py` читает граф через `astream(stream_mode=["messages","values"])` и форвардит токены **только узла `generate`** (узел `contextualize` тоже зовёт LLM — его поток фильтруем). Цитаты берём из финального снапшота `values`. Проверка: **56 токен-чанков по одному** (`In`→` hybrid`→` search`→`,`→`R`→`RF`), а не 1 блоб.
> Урок про неудачную попытку: первый заход (`stream_mode="messages"` + узел на `ainvoke`, `streaming=False`) дал **1 чанк** — LangGraph стримит токены, только если **сам узел стримит модель**. Поэтому перешли на `llm.astream` внутри узла.

**№8 — structured output в планнере.** Было: ручное срезание ```-заборов + `json.loads` + петля ретраев. Стало: `get_llm().with_structured_output(PlanModel)` — модель принуждается вернуть валидный объект (Pydantic-схема `PlanModel/TopicModel/SubtopicModel`), парсинг и ретраи удалены, граф линеен (`generate → persist`), есть fallback на минимальный план. Проверка: план генерится (HTTP 201, 5 топиков с часами/субтопиками).

### Проверка на данных
- Контекстуализация, стриминг (56 чанков), structured-план (201) — все зелёные.
- Полный набор тестов (chunker 4 + sparse 4 + reranker 2) — **10/10**.

### Вершина (опционально, не делали) — настоящий tool-calling агент
Сейчас mentor всё ещё фиксированный граф: **всегда** ищет ровно один раз. Можно дать LLM `retrieve` как **инструмент** и позволить ей самой решать: нужен ли поиск (на «привет» — нет), какой запрос послать, сходить ли в базу несколько раз (multi-hop для составных вопросов), когда хватит контекста. В LangGraph это паттерн `create_react_agent` / цикл `agent ⇄ tools` с условным ребром. Это следующий уровень — обсудим/сделаем при желании.

## Модуль 5 — Eval-харнес для ретрива (Трек A, шаг 1)

### Концепция
- До сих пор все рычаги (размер чанка, `rerank_candidates`, dense/sparse/rerank) настраивались **на глаз**. Это блокер: нельзя сказать, помог ли HyDE или late-chunking, если нет числа «до/после».
- **Golden-набор** — пары «вопрос → правильный чанк». На нём считаем метрики ранжирования:
  - **Hit@k** — попал ли нужный чанк в top-k (нашли вообще?).
  - **Recall@k** — доля правильных чанков в top-k.
  - **MRR** — `1/ранг` первого правильного (награда за позицию 1, а не 5).
  - **nDCG@k** — кредит каждому правильному по позиции, нормировано на идеал (0..1).

### Проблема, которую решал дизайн набора
- **`chunk_id` — это UUID, пересоздаётся при каждом re-ingest.** Если зашить сырые UUID в набор, он протухнет после первой же переиндексации. А трек C — это сплошные re-ingest (новый чанкер, parent-document, ColBERT).
- **Решение — стабильные якоря, а не id.** Вопрос указывает на чанк через `file` + уникальный сниппет текста (`gold`). `resolve.py` на каждом прогоне находит, в каких чанках Qdrant сейчас лежит этот текст. Набор привязан к **смыслу**, переживает любой re-ingest. Матчинг нормализован (lower + свёртка пробелов) — устойчив к шуму markdown.
- Формат — **TOML** (`goldset.toml`): stdlib `tomllib` (без зависимостей), комментарии, нативный UTF-8, удобно править руками.

### Что построено (`backend/app/eval/`)
- `metrics.py` — чистые функции метрик (покрыты `tests/test_eval_metrics.py`, 8/8 зелёных, значения посчитаны руками).
- `resolve.py` — загрузка набора + scroll payload Qdrant + резолв якорей → `chunk_id` + режим валидации (показывает, в какой чанк попал каждый якорь; ⚠ если 0 или слишком много матчей).
- `retrievers.py` — 4 стратегии → ранжированные `chunk_id`: `dense` / `sparse` (диагностические бэйзлайны, прямой `query_points`) + `hybrid` / `hybrid+rerank` (реальный `hybrid_search`, прод-путь).
- `run_eval.py` — каждая стратегия запрашивается **один раз** на максимальную глубину, все @k — срез одного списка (стратегии сравниваются на идентичных выборках). Таблица + снапшот в `results/<name>.json` для «до/после».
- Набор: **31 вопрос** по реальному документу `interviev_short.md` (Python-backend interview, укр.), все 6 разделов, микс `keyword`/`semantic`/`mixed` формулировок (чтобы видеть разницу dense vs sparse).

## Модуль 6 — Многоязычные модели (Трек C, эксперимент 1)

### Концепция
- Модуль 5 измерил конкретную причину слабого поиска: модели **англоцентричны**, а корпус **украинский**. dense (`all-MiniLM-L6-v2`) и reranker (`ms-marco-MiniLM`) не понимали кириллицу.
- Гипотеза: заменить обе на **многоязычные** → dense должен взлететь, reranker — перестать вредить. Проверяем числом через тот же харнес (Трек A).
- Выбран стек: эмбеддер **`intfloat/multilingual-e5-base`** (768d, лимит 512), reranker **`cross-encoder/mmarco-mMiniLMv2-L12-H384-v1`**. Легче на CPU, чем `bge-m3`/`bge-reranker-v2-m3`, и учит нюанс префиксов.

### Проблема в нашем коде (3 нетривиальных места)
1. **Префиксы e5.** e5 обучен с инструкциями: запрос кодируется как `query: <текст>`, документ — `passage: <текст>`. **Асимметрично.** Без префиксов качество заметно проседает. → развёл пути: `embed_query` (`query: `) vs `embed_passages` (`passage: `), сами префиксы в `config.py` (для prefix-free моделей = `""`).
2. **Смена размерности → пересоздание коллекции.** 384d→768d: Qdrant-коллекция жёстко привязана к размерности. `lifespan` создаёт коллекцию только если её нет → недостаточно рестарта. Написал `app/eval/reingest.py`: дропает + создаёт коллекцию под текущую размерность, переиндексирует **ровно тот же корпус** (те же `document_id`, что были в коллекции) новым токенайзером + passage-префиксом. Пин корпуса через `--filenames` — для воспроизводимого «до/после».
3. **`EMBEDDING_MODEL` в `docker-compose.yml` молча перекрывал `config.py`.** Pydantic-settings читает env поверх дефолтов. После правки `config.py` прогон всё ещё грузил MiniLM — а reranker уже сменился (его в env не было). → убрал хардкод из compose: модель/размерность/префиксы теперь **только в `config.py`** (единый источник, не могут разойтись).

### Инфра-засада: память
- У Docker всего **~2 ГБ** RAM. e5-base крупнее MiniLM; uvicorn с e5+mmarco ≈ 1.8 ГБ. Второй процесс (`exec` reingest/eval) грузил **второй экземпляр e5** → OOM-kill, контейнер перезапускался (RestartCount рос).
- Решение: модель-процессы гонять **по одному** — `docker compose stop backend`, затем reingest/eval как standalone `docker compose run --rm --no-deps` (один экземпляр модели в RAM), потом `start backend`. Плюс **named-volume `hf_cache`** на `/root/.cache/huggingface` → standalone-контейнеры переиспользуют скачанные модели, не качают заново.

### Проверка на данных — гипотезы подтвердились с запасом
Тот же набор (31 вопрос), тот же корпус (3 документа), сменился только энкодер-стек:

| стратегия | Hit@5 (baseline→multi) | MRR (baseline→multi) | nDCG@5 |
|---|---|---|---|
| dense | 0.548 → **0.968** | 0.388 → **0.812** | 0.394 → 0.835 |
| sparse | 0.806 → 0.839 | 0.624 → 0.679 | 0.648 → 0.701 |
| hybrid | 0.806 → 0.968 | 0.612 → 0.798 | 0.640 → 0.834 |
| hybrid+rerank | 0.645 → **1.000** | 0.526 → **0.850** | 0.521 → 0.880 |

**Три измеренных факта:**
1. **dense взлетел**: Hit@5 0.55 → **0.97** (+0.42), MRR 0.39 → **0.81**. Из худшей стратегии — в одну из лучших. Многоязычный вектор ловит укр. семантику.
2. **reranker из вредителя стал лучшим**: в Модуле 5 он ронял Hit@5 0.81→0.65; теперь `hybrid+rerank` — **топ** (Hit@5 1.00, MRR 0.85, nDCG@5 0.88). Многоязычный cross-encoder скорит укр. пары правильно. → `rerank_enabled=True` теперь подтверждён **числом**, а не дефолтом.

### Честный конфаунд: переразбивка
- Токенайзер e5 (SentencePiece/XLM-R) кодирует кириллицу **нативно**, а WordPiece у MiniLM её сильно дробит. При том же лимите ~240 токенов e5 вмещает больше текста → `interviev_short.md` переразбился на **81 чанк вместо 108**. Т.е. сравниваем «энкодер + его токенизация» (по-другому e5 не используешь).
- Но магнитуда (+0.42 dense) на порядок больше, чем мог бы дать сам чанкинг, а языконезависимый sparse едва дрогнул → конфаунд не объясняет результат. Полностью изолировать можно было бы, пере-сняв baseline на 81-чанковой раскладке MiniLM — оставлено как опция.
- Якоря набора (`file`+сниппет) пережили переразбивку: **31/31** резолвятся чисто. Ровно ради этого Модуль 5 строил набор на якорях, а не на UUID.

> **Главный урок Модуля 6:** провал dense/reranker из Модуля 5 был не «техника плохая», а «модель не на том языке». Точечная замена на многоязычный стек, **измеренная тем же харнесом**, развернула картину: dense и rerank из аутсайдеров — в лидеры. Сначала измерь причину (Трек A), потом чини (Трек C) — и снова измерь.

## Модуль 7 — HyDE (Трек C, эксперимент 2)

### Концепция
- **HyDE** (Hypothetical Document Embeddings): короткий вопрос («що таке RRF?») сформулирован как *вопрос*, а пассажи-ответы — как *утверждения*; в векторном пространстве это разные кластеры. Идея: пусть LLM сначала напишет **гипотетический ответ-пассаж**, и мы ищем по эмбеддингу этого выдуманного текста — он лежит в «пространстве ответов», ближе к реальным пассажам. Даже фактически кривой черновик помогает геометрически.
- Гипотеза по эффект/цена (роадмап): дёшево (1 LLM-вызов перед поиском), узел перед retrieve, re-ingest не нужен (меняется **только query-сторона**, корпус и эмбеддинги пассажей не трогаются).
- **Предсказание до прогона:** HyDE окупается, когда энкодер слаб или запрос терсный/размытый. У нас после Модуля 6 dense уже у потолка (Hit@10=1.00) → хедрума мало, ждём в лучшем случае ноль, в худшем — дрейф. Проверяем числом.

### Что построено
- `app/services/rag/hyde.py` — `generate_hypothetical(query)`: LLM (temp=0 для детерминизма) пишет 2–4 предложения **на языке вопроса** (укр. вопрос → укр. черновик; англ. черновик попал бы не в тот языковой кластер мультиязычного e5). **Graceful degradation:** пустой вывод / ошибка LLM → fallback на исходный запрос, так что HyDE никогда не хуже no-HyDE из-за сбоя.
- `retriever.py`: `hybrid_search(..., hyde_doc=None)`. Dense-вектор берётся из гипо-дока; **sparse остаётся на исходном запросе** — длинный сгенерённый абзац размыл бы точные ключевые слова, на которых держится BM25. Reranker тоже скорит по **исходному** запросу. Флаг `hyde_enabled` (config, по умолч. `False`).
- `config.py`: `hyde_enabled: bool = False`. `llm_factory.py`: параметр `temperature`.
- Eval: стратегии `dense+hyde`, `hybrid+hyde`, `hybrid+hyde+rerank`. **Один черновик на вопрос** через мемо `_hypo` — три HyDE-стратегии сравниваются на идентичном гипо-доке (и платим 1 LLM-вызов на вопрос, не 3). Тесты HyDE — 5 шт. (стаб LLM, без сети), всего 23/23.

### Проверка на данных — гипотеза подтвердилась: HyDE здесь не помогает
Тот же набор (31 вопрос), та же коллекция e5 из Модуля 6, без re-ingest. До = `multilingual.json`, после = `hyde.json`:

| стратегия | Hit@1 | Hit@5 | MRR@10 | nDCG@5 |
|---|---|---|---|---|
| dense | 0.677 | 0.968 | 0.812 | 0.835 |
| **dense+hyde** | 0.613 ↓ | 0.903 ↓ | 0.747 ↓ | 0.776 ↓ |
| hybrid | 0.645 | 0.968 | 0.785 | 0.824 |
| **hybrid+hyde** | 0.581 ↓ | **1.000 ↑** | 0.764 ↓ | 0.829 ≈ |
| hybrid+rerank | 0.742 | 1.000 | 0.850 | 0.880 |
| **hybrid+hyde+rerank** | 0.742 = | 1.000 = | 0.850 = | 0.880 = |

**Три измеренных факта:**
1. **dense+hyde строго хуже** по всем метрикам (Hit@5 0.97→0.90, MRR 0.81→0.75). Подтверждает предсказание: dense у потолка, спасать нечего, а гипо-абзац вносит **дрейф** — нужный чанк опускается на ранг ниже (отсюда просадка Hit@1/MRR при ещё неплохом Hit@10).
2. **hybrid+hyde** — единственный плюс: Hit@5 0.968→**1.000** (закрыл 1 промах). Но ценой ранней точности — Hit@1 0.645→0.581, MRR 0.785→0.764. Нетто **не выигрыш**: лучше найти на месте 1, чем на месте 5.
3. **hybrid+hyde+rerank ≡ hybrid+rerank** — **побайтово** одинаковы (0.742 / 1.000 / 0.850 / 0.880). Reranker берёт top-20 кандидатов и переупорядочивает по **исходному** запросу; пока нужный чанк попал в пул (а он попадает в обоих случаях), финальный порядок не зависит от того, HyDE его поднял или RRF. → **сильная стадия-2 нивелирует выбор метода стадии-1.**

### Нюанс, отмеченный честно (не тестировал отдельно)
- **Префикс гипо-дока.** e5 асимметричен: `query:` vs `passage:`. Я эмбеддил гипо-док с `query:` (он всё ещё *search-сторона* пары). Логичная альтернатива — `passage:` (гипо-док маскируется под документ → теоретически ближе к реальным пассажам). Не прогонял отдельной колонкой, чтобы не раздувать таблицу; оставлено как открытая переменная. Учитывая, что HyDE и так не дал нетто-плюса, едва ли префикс развернул бы вывод.

> **Главный урок Модуля 7:** «дёшево и популярно» ≠ «полезно на твоих данных». HyDE — заслуженно известная техника, но окупается в конкретном режиме (слабый энкодер / размытый запрос), а наш — обратный. Харнес сэкономил от включения фичи, которая **тихо просадила бы Hit@1/MRR в проде**. `hyde_enabled=False` — подтверждено числом. Бонус-инсайт: **сильный reranker делает выбор ретривера 1-й стадии безразличным** (пока кандидат в пуле) — значит силу стоит вкладывать в recall пула, а не в дорогую переформулировку запроса.

## Модуль 8 — Умный чанкинг (Трек C, эксперимент 3)

### Концепция
- До сих пор секцию резали **скользящим окном по сырым token-id** (`_split_by_tokens`): окно из N токенов, шаг `N − overlap`, decode обратно в текст. Граница чанка падает на **произвольное смещение** — посреди предложения, а то и посреди слова. Ответ на вопрос может оказаться **разорван** между двумя чанками, и ни в одном нет мысли целиком.
- **Рекурсивный сплиттер** (идея LangChain `RecursiveCharacterTextSplitter`, но token-aware): резать по **самой крупной естественной границе, которая влезает** — абзац (`\n\n`) → строка (`\n`) → предложение (`.!?…`), затем **жадно паковать** целые единицы до бюджета токенов. Фразу рвём только если **одно предложение** превышает лимит модели — там единственный безопасный путь это token-окно (фолбэк сохранён → жёсткий cap не нарушен).
- Overlap тоже стал «умным»: переносим в следующий чанк **целые хвостовые предложения** (сумма ≤ `chunk_overlap`), а не обрезок токенов. И граница, и перекрытие теперь лежат на краях предложений.
- **Предсказание до прогона:** dense/hybrid уже у потолка (Hit@5 0.97, rerank 1.00) после Модуля 6 — хедрума почти нет. Голд-сниппеты короткие и сидят **внутри** чанков, а не на границах, которые двигает новый сплиттер. Ждём в лучшем случае нулевой Δ метрик при **качественно более чистых** чанках. Проверяем числом.

### Что построено
- `chunker.py`: `_atomize` (рекурсивный разбор по `_SEPARATORS`, гарантирует единицы ≤ max_tok) + `_pack` (жадная упаковка с overlap целыми единицами; единицы джойнятся одним пробелом, чтобы re-tokenize джойна ≤ суммы по-юнитно — SentencePiece сливает ведущий пробел, лимит держится) + `_split_section` (диспетчер по флагу). Точки входа `chunk_markdown`/`chunk_pdf_pages` теперь зовут `_split_section`.
- `config.py`: флаг `chunking_strategy: "recursive" | "token"` (дефолт `recursive`). `token` — легаси-окно Модулей 1–7, оставлен для воспроизводимого «до/после».
- Тесты: +3 регрессии (предложения не рвутся, overlap несёт целое предложение, лимит модели соблюдён рекурсивным путём). Старые 4 теста чанкера зелёные (включая «нет границ → фолбэк на token-окно покрывает всё»). Всего **26/26**.
- Это **требует re-ingest** (меняется раскладка чанков) — в отличие от HyDE. Корпус пин `--filenames`, якоря goldset (`file`+сниппет) пережили переразбивку: **31/31** резолвятся чисто.

### Проверка на данных — гипотеза подтвердилась: на этом корпусе чанкинг не сдвинул метрики
Тот же набор (31 вопрос), тот же e5-стек, сменилась **только раскладка чанков**. До = `multilingual.json` (token-окно), после = `smart_chunk.json` (recursive). Прод-стратегии:

| стратегия | Hit@1 | Hit@5 | MRR@10 | nDCG@5 | Δ |
|---|---|---|---|---|---|
| dense | 0.677 → 0.677 | 0.968 → 0.968 | 0.812 → 0.812 | 0.835 → 0.835 | 0 |
| sparse | 0.548 → 0.548 | 0.839 → 0.839 | 0.679 → 0.679 | 0.701 → 0.701 | 0 |
| hybrid | 0.677 → 0.677 | 0.968 → 0.968 | 0.798 → 0.798* | 0.834 → 0.834 | ~0 |
| hybrid+rerank | 0.742 → 0.742 | 1.000 → 1.000 | 0.850 → 0.850 | 0.880 → 0.880 | 0 |

\* Единственное движение во всей таблице — на 3–4-м знаке и **только на глубине @10**: hybrid MRR 0.7976→0.7984, hybrid nDCG@10 0.8451→0.8458, sparse nDCG@10 0.7301→0.7304. Всё на @1/@3/@5 — **заморожено**. Это шум одного чанка, сместившегося на ранг где-то у позиции 9–10.

**Три измеренных/проверенных факта:**
1. **Δ ≈ 0 по всем стратегиям.** Ни одна метрика не сдвинулась даже на 1/31 (0.032) на @1/@3/@5. Рекурсивный чанкинг **не улучшил и не ухудшил** ретрив на этом корпусе.
2. **Чанки реально изменились** (не ложный ноль): `interviev_short.md` дал те же **81 чанк**, но границы теперь на концах предложений — проверено по содержимому (`…інструкцію за інструкцією.`, `…стекову машину.`, `…добре працює для I/O.`); token-окно резало бы посреди слова. Раскладка сменилась — метрики нет.
3. **Качественный выигрыш, невидимый харнесу.** Вопрос `acid` (gold `atomicity`+`durability`) в Модуле 5 **спанил два чанка** — теперь определение ACID целиком в одном чанке (resolve: 1 id вместо 2). Для recall это ничего не дало (его и так находили), но в LLM-контекст уходит цельная мысль, а не обрубок.

### Почему ноль (разбор)
- **Гранулярность не менялась** (81 чанк, тот же бюджет 256/overlap 48) — двигались только пара токенов на каждой границе: кусок предложения переезжает из конца чанка N в начало N+1. Ядро чанка ~на 95% то же → dense-вектор (768d) едва шевелится, BM25 по ядру не меняется.
- **Метрики у потолка** — спасать нечего.
- **Голд-термины сидят внутри чанков, а не на границах**, которые релоцирует сплиттер. Ровно та зона, где «не рвать фразу» помогло бы (термин на стыке), в наборе почти не представлена.

### Честные конфаунды
- Baseline `multilingual.json` снят в Модуле 6 тем же **token-алгоритмом** на том же корпусе/энкодере — это валидный «до». Свежий token-снимок на текущем коде не пере-снимал (детерминированный чанкинг + детерминированный эмбеддинг → отличие было бы нулевым; стоил бы ещё один OOM-цикл reingest+eval). Кто хочет строго изолировать — `chunking_strategy=token` + reingest воспроизводит baseline.
- Сравниваем «рекурсивный сплиттер + его раскладку» (по-другому его не использовать), но раз гранулярность (81 чанк) и энкодер те же, переменная изолирована лучше, чем в Модуле 6.

### Решение по дефолту
- Оставил `chunking_strategy="recursive"` **включённым** (в отличие от HyDE, который выключил). Логика: для ретрива нейтрален (Δ≈0, не вредит — в отличие от HyDE, тихо ронявшего Hit@1), **бесплатен** при ингесте (чистые CPU-строки, детерминирован, без LLM/доп.модели), и даёт качественный плюс — чанки на границах предложений → чище контекст для LLM, определения не разорваны. Флаг позволяет откатиться на `token`.

> **Главный урок Модуля 8:** «чище инженерно» ≠ «лучше по метрике» — и это нормальный, измеренный результат, а не провал. Рекурсивный чанкинг объективно правильнее (не рвёт фразы, склеил разорванный ACID), но на корпусе у потолка с короткими голд-терминами внутри чанков ретрив-метрика этого не видит. Решение оставить фичу принято **не по метрике** (она ноль), а по соотношению «нулевой риск + бесплатно + качественный плюс для downstream-LLM». Второй раз подряд (после HyDE) харнес показал, что популярная техника на наших данных метрику не двигает — но здесь, в отличие от HyDE, у изменения есть бесплатный upside, поэтому вердикт обратный: HyDE off, recursive on. **Semantic chunking** (граница по скачку смысла, требует эмбеддить каждое предложение) — отдельный эксперимент (Модуль 9), отложен: тяжелее по памяти (Docker ~2 ГБ) и это уже другой метод (дисциплина «один за раз»).

## Модуль 9 — Semantic chunking (Трек C, эксперимент 4)

### Концепция
- Рекурсивный сплиттер (Модуль 8) режет по **синтаксическим** границам (абзац→строка→предложение) и жадно пакует до бюджета токенов — граница падает там, где **кончается бюджет**, вслепую к смыслу. **Semantic chunking** режет там, где **прыгает смысл**: эмбеддим каждое предложение, меряем косинус-расстояние между соседями, ставим границу на **самых больших скачках** (топ-(100−P) перцентиль). Идея Greg Kamradt / LlamaIndex `SemanticSplitterNodeParser`.
- **Предсказание до прогона (из Модуля 8):** метрики у потолка (dense/hybrid Hit@5≈0.97, rerank=1.00), голд-термины сидят внутри чанков → если гранулярность не изменится, ждём Δ≈0, как в Модуле 8. **НО** честно зафиксировали главный риск: semantic-границы могут **сменить число/размер чанков** — и тогда эффект будет уже не от «умных границ», а от гранулярности. Этот конфаунд и выстрелил.

### Что построено
- `chunker.py`: `_sentence_units` (всегда дробит на предложения — в отличие от `_atomize`, который дробит только при превышении лимита; semantic обязан сравнивать соседей даже когда секция влезает в бюджет) + `_semantic_split` (эмбед каждого юнита через **синглтон** `get_embedder()`/`embed_passages`, косинус-дистанции соседей, разрез на `>=` перцентиль-порога, затем `_pack` каждого сегмента под токен-cap) + `_percentile` (numpy-free, линейная интерполяция). Диспетчер `_split_section` получил ветку `semantic`.
- `config.py`: `chunking_strategy` теперь `recursive | semantic | token`; новый `semantic_breakpoint_percentile: float = 95.0`.
- Тесты: +4 (`_percentile` против numpy-семантики, рез на стыке двух тем, токен-cap соблюдён, single-unit passthrough без лишнего эмбеда). Всего **30/30**.
- Требует re-ingest (меняется раскладка). Якоря goldset пережили переразбивку: **31/31** резолвятся чисто (часть теперь на 2 чанка — мельче чанки + overlap).

### Проверка на данных — гипотеза о конфаунде подтвердилась: semantic заметно ПРОСАДИЛ ретрив
Тот же набор (31 вопрос), тот же e5-стек, сменилась **только раскладка чанков**. До = `smart_chunk.json` (recursive, 81+7+5 = **93 чанка**), после = `semantic_chunk.json` (semantic@P95, 158+12+10 = **180 чанков**, почти ×2). Прод-стратегии:

| стратегия | Hit@1 | Hit@5 | MRR@10 | nDCG@5 | Δ Hit@1 |
|---|---|---|---|---|---|
| dense | 0.677 → 0.484 | 0.968 → 0.935 | 0.812 → 0.685 | 0.835 → 0.721 | **−0.19** |
| sparse | 0.548 → 0.387 | 0.839 → 0.774 | 0.679 → 0.544 | 0.701 → 0.577 | **−0.16** |
| hybrid | 0.677 → 0.548 | 0.968 → 0.871 | 0.798 → 0.699 | 0.834 → 0.718 | **−0.13** |
| hybrid+rerank | 0.742 → 0.419 | 1.000 → 0.935 | 0.850 → 0.627 | 0.880 → 0.672 | **−0.32** |

**Три измеренных факта:**
1. **Регрессия по всем стратегиям и всем @k.** Сильнее всего бьёт по Hit@1 и MRR (ответ скатывается с 1-го места вниз). Прод-путь `hybrid+rerank` рухнул с **идеального** (Hit@5=1.00, Hit@1=0.74, MRR=0.85) до Hit@1=0.42 / MRR=0.63 — −0.32 по Hit@1.
2. **Гранулярность взорвалась:** 93 → 180 чанков (×1.9 по каждому файлу: interviev 81→158, test 7→12, rerank_demo 5→10). Чанки стали вдвое мельче → контекст фрагментирован, «правильный» чанк держит меньше сигнала, конкурирует с роем похожих мелких соседей. Reranker пострадал сильнее всех: на обрубках cross-encoder’у нечего взвешивать.
3. **Перцентиль-ручка инертна на этом корпусе.** Прогон с `semantic_breakpoint_percentile=98` дал **идентичные 158 чанков** (count-verified). Причина: корпус — интервью из множества **коротких Q&A-секций**; для секции с малым числом предложений 95-й и 98-й перцентили линейно интерполируются почти к максимуму, и `>=`-порог форсирует **один и тот же** рез по максимуму дистанции в каждой секции. Раскладка идентична → eval не пере-прогонял (числа были бы те же).

### Почему просадка (разбор)
- **Корень — мой `>=`-дизайн + отсутствие слияния мелких сегментов.** `>=` порог я выбрал нарочно («секция всегда режется по сильнейшему скачку, не схлопывается в один гигантский чанк»). Но на корпусе коротких секций это значит: **каждая** многопредложенческая секция получает ≥1 форс-рез, а `_pack` мелкие сегменты обратно не сливает (в отличие от жадной упаковки recursive до 240 токенов). Итог — почти удвоение числа чанков.
- **Финер ≠ лучше.** Контраст с Модулем 8 кристально чист: там гранулярность НЕ менялась (81 чанк) → Δ≈0; здесь она удвоилась → −0.13…−0.32. Харнес **видит** чанкинг ровно тогда, когда тот двигает гранулярность, и для нашего корпуса/потолка более мелкие чанки строго хуже.
- **Идея «умных границ» здесь не проверена в чистом виде** — её эффект перекрыт конфаундом гранулярности (что и предсказывали). Чтобы изолировать, нужен другой дизайн порога: **глобальный/абсолютный** (может дать 0 резов в коротких когерентных секциях) либо **min-chunk-size merge** мелких сегментов до бюджета. Это уже другой метод → отдельный эксперимент (дисциплина «один за раз»).

### Честные конфаунды
- Сравниваем «semantic + его раскладку 180 чанков» против «recursive + 93 чанка» — переменная **не** изолирована: меняются и границы, и гранулярность одновременно (зафиксировано как риск ещё до прогона). Поэтому вывод корректно звучит так: «semantic chunking *в этой реализации, на этом корпусе* переизмельчает и роняет ретрив», а **не** «семантические границы вредны в принципе».
- baseline `smart_chunk.json` снят в Модуле 8 тем же e5-стеком на том же корпусе — валидный «до».

### Решение по дефолту
- `chunking_strategy="recursive"` **остаётся дефолтом**. Semantic — за флагом, выключен (как HyDE): он не нейтрален, а **строго хуже** (−0.32 Hit@1 на прод-пути) при вдвое большем числе чанков. Код сохранён под `semantic` для воспроизводимости и будущего эксперимента с другим порогом.

> **Главный урок Модуля 9:** конфаунд, названный вслух **до** прогона, выстрелил — и это победа метода, а не провал. Гипотеза Модуля 8 гласила «Δ≈0, ЕСЛИ не изменится гранулярность»; semantic изменил её вдвое → −0.13…−0.32. Три эксперимента подряд харнес выносит вердикт, который на глаз не увидеть: HyDE (тихо вредит → off), recursive (Δ≈0 + бесплатный upside → on), semantic (явно вредит переизмельчением → off). И главное methodological: «популярная SOTA-техника» (semantic chunking — модный дефолт в туториалах) на конкретном корпусе оказалась **строго хуже** простого рекурсивного сплиттера. Без харнеса её бы включили «потому что умнее звучит». Меряем — не угадываем.

## Модуль 10 — Late chunking (Трек C, эксперимент 5)

### Концепция
- Все стратегии до сих пор эмбеддили **каждый чанк изолированно**: вектор чанка видел только токены самого чанка. **Late chunking** (Günther et al., Jina AI, 2024) переворачивает порядок: сначала прогнать через энкодер **весь документ** (каждый токен контекстуализируется всем документом), **потом** усреднить token-level эмбеддинги по спанам чанков. Вектор чанка тогда «помнит» соседний текст (местоимение разрешается, голое определение наследует субъект) — **не дробя мельче** (провал semantic chunking, Модуль 9).
- **Главная сложность, названная до прогона:** e5 читает максимум `max_seq_length=512` токенов. Документ длиннее (interviev_short body = **6553 токена**) **нельзя** прогнать одним окном — буквальная предпосылка late chunking на коротко-контекстном энкодере **недоступна**. Аппроксимируем **скользящим окном по токенам** (passage-префикс на каждое окно, усреднение на перекрытии 64 токена). Это центральный, честно зафиксированный конфаунд: контекстуализация **локальная** (шириной в одно окно), а не глобальная.
- **Предсказание до прогона:** метрики у потолка; корпус — короткие самодостаточные Q&A-секции (большинство чанков и так находятся); окно 512 не доносит «весь документ»; плюс пулинг асимметричен запросу. Жду Δ≈0 или **слегка отрицательно** на dense (пулинг чанк-контента без префикса/спец-токенов — другое распределение, чем naive `embed_query`), при нейтральном прод-пути (сильный reranker поглотит, как в Модуле 7).

### Что построено
- `embedder.py`: `embed_spans_late(body, spans)` — токенизирует весь body с offset-mapping, гонит **скользящие окна ≤512** через `auto_model` (token-level `last_hidden_state`), усредняет эмбеддинги на перекрытии окон, затем mean-pool токенов каждого спана + L2-норма. Пулит **только content-токены** спана (без `[CLS]`/префикса/`[SEP]`) — это и есть формулировка late chunking. Доступ к HF-энкодеру через `get_embedder()[0].auto_model` (а не `SentenceTransformer.encode`, который прячет токен-уровень за своим пулингом).
- `chunker.py`: `_unit_spans` (восстанавливает char-offset каждой атомарной единицы — переиспользует `_atomize`, поэтому **гранулярность та же, что у recursive**) + `_pack_spans` (жадная упаковка единиц в спаны до бюджета, **без overlap** — у late чистый шов, контекст несёт сам документ-эмбеддинг, а не повтор соседей) + `_late_chunk_sections` (склеивает секции в один document body, режет на спаны, зовёт `embed_spans_late`, кладёт вектор в `Chunk.dense_vector`).
- `Chunk.dense_vector: list[float] | None` — late пред-вычисляет вектор (его нельзя пересобрать из текста чанка); `upsert_chunks` использует готовый вектор, если он есть, иначе эмбеддит текст как обычно. Прочие стратегии не тронуты.
- `config.py`: `chunking_strategy` теперь `recursive | semantic | late | token`; новый `late_chunk_window_overlap: int = 64`.
- Тесты: +3 (offset-спаны восстанавливают verbatim-подстроку; спаны контигуальны без overlap и в пределах cap; late кладёт по 1 вектору на чанк, текст не теряется — heavy forward застаблен). Всего **33/33**.
- Требует re-ingest. Якоря goldset пережили: **31/31** резолвятся чисто.

### Проверка на данных — гипотеза подтвердилась: late chunking ПРОСАЖИВАЕТ dense, прод-путь нейтрален
Тот же набор (31 вопрос), тот же e5-стек. **Гранулярность не изменилась: 93 чанка — РОВНО как у `smart_chunk` (recursive)** (interviev 81, test 7, rerank_demo 5). До = `smart_chunk.json`, после = `late_chunk.json`:

| стратегия | Hit@1 | Hit@5 | MRR@10 | nDCG@5 |
|---|---|---|---|---|
| dense | 0.677 → **0.419** | 0.968 → 0.806 | 0.812 → **0.591** | 0.835 → 0.624 |
| sparse | 0.548 → 0.548 | 0.839 → 0.839 | 0.679 → 0.678 | 0.701 → 0.701 |
| hybrid | 0.677 → 0.516 | 0.968 → **1.000** | 0.798 → 0.690 | 0.834 → 0.757 |
| hybrid+rerank | 0.742 → **0.742** | 1.000 → 1.000 | 0.850 → 0.852 | 0.880 → 0.882 |

**Четыре измеренных факта:**
1. **sparse заморожен побайтово** (0.548/0.839/0.678/0.701). BM25 не трогает dense-векторы, а гранулярность та же (93 чанка) → **чище изоляции в треке не было**: единственная изменённая переменная — сам метод эмбеддинга dense. Не как в Модуле 9, где гранулярность взорвалась и маскировала эффект.
2. **dense просел сильно:** Hit@1 −0.26, Hit@5 −0.16, MRR −0.22, nDCG@5 −0.21. Late-пулинг даёт **строго худшие** dense-векторы, чем naive per-chunk e5.
3. **hybrid — рано хуже, глубоко ок:** Hit@1 −0.16 (dense тянет вниз 1-е место), но Hit@5 0.968→**1.000** (sparse+fusion вытягивают к 5-му). Ранняя точность страдает, recall пула — нет.
4. **hybrid+rerank ≡ baseline** (Hit@1 0.742=0.742, Hit@5 1.00=1.00; MRR/nDCG +0.002 — шум). Cross-encoder **полностью поглощает** деградацию dense, т.к. пул (hybrid Hit@5=1.0) всё ещё содержит голд-чанк. **Повтор инсайта Модуля 7:** сильная стадия-2 нивелирует выбор стадии-1, пока кандидат в пуле.

### Почему просадка (разбор)
- **Локальная контекстуализация (тот самый конфаунд).** Окно 512 ≪ документ (6553) → «весь документ» в вектор не попадает; чанк получает контекст в лучшем случае пары соседних секций. На корпусе **коротких самодостаточных** чанков (~80 токенов в среднем) этот чужой контекст не добавляет сигнала, а **размывает** собственную тему чанка.
- **Асимметрия пулинга = train/inference mismatch.** Late пулит только content-токены пассажа (без префикса/спец-токенов, под влиянием соседнего текста), а запрос идёт через naive `embed_query` («query: » + mean-pool по всей короткой строке). e5 обучен на паре «независимо mean-pooled query / passage **с** префиксом+спецтокенами». Late-вектор лежит в **другом распределении** → геометрия query↔passage плывёт → dense-матч хуже.
- **Late chunking — для длинноконтекстных энкодеров.** Оригинал Jina — на модели с окном 8192, где чанк реально видит весь документ И энкодер устойчив к span-пулингу. На e5 (512) обе предпосылки нарушены. **Техника не «плохая» — она вне своего режима.**

### Честные конфаунды
- Сравниваем «late + его 93 спана» против «recursive + 93 чанка» — **гранулярность совпала** (в отличие от Модуля 9), но late **убрал overlap** (чистые швы). На этом корпусе overlap почти не срабатывал (секции короче бюджета) → число чанков не изменилось; эффект overlap на этих данных ≈ ноль, так что переменная изолирована хорошо.
- Окно 512 + усреднение перекрытий — аппроксимация «одного прохода», не он сам. На энкодере с настоящим длинным окном вывод мог бы отличаться (это и есть граница применимости, не баг реализации).

### Решение по дефолту
- `chunking_strategy="recursive"` **остаётся дефолтом**. Late — за флагом, **выключен** (как HyDE и semantic): он не нейтрален, а **строго хуже на dense/hybrid-раннем-ранге** при нулевом плюсе на прод-пути с reranker. Код сохранён под `late` для воспроизводимости и для будущего прогона на длинноконтекстном энкодере (где у техники есть шанс).

> **Главный урок Модуля 10:** четвёртый эксперимент подряд харнес выносит контринтуитивный вердикт — и впервые с **идеальной изоляцией** (sparse заморожен, гранулярность 93=93). Late chunking — «умнее звучит, чем recursive», но на коротко-контекстном e5 + корпусе коротких секций он строго просаживает dense (−0.26 Hit@1), потому что (а) окно 512 физически не доносит «весь документ» и (б) span-пулинг рассинхронен с naive-запросом. Главное methodological: **техника окупается только в своём режиме** (длинный контекст + энкодер, обученный/устойчивый к span-пулингу) — как HyDE окупается при слабом энкодере, а здесь — нет. И снова: **сильный reranker делает выбор метода эмбеддинга стадии-1 безразличным** (прод-путь не дрогнул). Карта трека C по чанкингу/ретриву теперь: recursive on (Δ≈0, бесплатный upside), HyDE/semantic/late — off (измеренно хуже или бесполезны). Сила — в recall пула и reranker'е, не в «умном» одиночном векторе.

## Модуль 11 — Parent-document retrieval (Трек C, эксперимент 6)

### Концепция
- Идея: искать по **мелким** чанкам (точность), а в LLM отдавать **родительский** крупный блок (полнота контекста). **Дизайн A:** индекс не трогаем — те же 93 recursive-чанка, **re-ingest не нужен**, раскрытие живёт **только на возвратном пути** `hybrid_search`. Два варианта родителя: **«секция»** (все чанки с тем же heading-path: `document_id` + h1/h2/h3, склейка по `chunk_index`) и **«окно соседей»** (`chunk_index ± N` в пределах документа). **Дедуп:** несколько хитов в один родитель → один блок (освобождает top-k слоты).
- **Что меряем — зафиксировано ДО прогона.** Ранжирование первичного хита харнес почти не двигает (ищем по тем же чанкам) → ждём **Δ≈0** на Hit@1/MRR. Единственный реально измеримый угол — **multi-chunk gold** (вопросы, чей gold размазан по 2 чанкам): раскрытие может поднять Recall@k, **если харнес зачтёт чанки-составляющие родителя**.
- **Решение по метрике:** block-aware scoring. Стратегия возвращает ранжированные **блоки** (каждый = множество покрытых `chunk_id`); метрики кредитуют **каждый** покрытый чанк. Block-метрики **точно редуцируются** к chunk-метрикам на singleton-блоках (доказано тестом) → все прежние снапшоты воспроизводятся побайтово, non-parent строки таблицы не меняются.

### Что построено
- `config.py`: `parent_retrieval` (`off | section | window`) + `parent_window=1`, **off** по умолчанию.
- `qdrant_store.fetch_document_chunks(document_id)` — соседи чанка по `document_id` (scroll + сортировка по `chunk_index`).
- `retriever._expand_to_parents` + параметр `parent` в `hybrid_search`; блок несёт `member_chunk_ids` + склеенный `content` (то, что прочтёт LLM), сохраняя первичный хит как «голову».
- `eval`: `block_hit/recall/rr/ndcg`, стратегии `parent-sec` / `parent-win`, нормализация «flat list → singleton-блоки» в `run_eval`.
- Тесты: +8 (block-редукция == chunk-метрики; section/window expansion; дедуп секции/окна; регрессия `chunk_index`). Всего **41/41**. Goldset **31/31**.

### Грабли по дороге — ловушка «зелёного» результата
- **Первый прогон: `parent-sec` И `parent-win` ≡ `hybrid+rerank` ПОБАЙТОВО.** Это идеально совпало с прогнозом «Δ≈0» — я чуть не записал «гипотеза подтвердилась». Инспекция per-question вскрыла: `parent-win` блок №1 для смежной пары gold `[33,34]` = `[33]` (один чанк), а не `[32,33,34]`. **Баг:** `hybrid_search` не клал `chunk_index` в результат-dict → окно схлопывалось в один чанк → `parent-win` был **не окном, а no-op**. Юнит-тест проходил, потому что звал `_expand_to_parents` с рукодельными dict'ами, где `chunk_index` был — зазор сидел на **интеграционном шве**. Фикс: положить `chunk_index` в payload-маппинг + регрессионный тест. **Урок: результат, совпавший с прогнозом, всё равно может быть прав по неверной причине — проверяй МЕХАНИЗМ, а не только метрику.**

### Проверка на данных — после фикса
Тот же набор (31 вопрос), recursive baseline (= `smart_chunk.json`), индекс не менялся. До = `hybrid+rerank`, после = `parent-*`:

| стратегия | Hit@1 | Recall@1 | Recall@3 | MRR@10 | nDCG@5 |
|---|---|---|---|---|---|
| hybrid+rerank | 0.742 | 0.677 | 0.935 | 0.850 | 0.880 |
| parent-sec | 0.742 | 0.677 | 0.935 | 0.850 | 0.880 |
| parent-win | **0.871** | **0.823** | 0.984 | **0.935** | 0.948 |

**Замер контекст-цены (chunks delivered на слот):**

| | k=1 | k=3 | k=5 |
|---|---|---|---|
| hybrid+rerank | 1.00 | 3.00 | 5.00 |
| parent-win | **3.00** | **8.48** | **13.90** |

**Два факта:**
1. **`parent-sec` — ТОЧНЫЙ no-op.** Каждая heading-секция корпуса = **ровно один чанк** (короткие Q&A-подсекции влезают в 256-tok бюджет recursive-чанкера) → «родительская секция» = сам чанк → раскрывать нечего. Диагностика подтвердила: каждый section-блок — singleton. parent-document **по секции** что-то делает только там, где секция растянута на несколько чанков (длинная проза) — **не наш корпус**.
2. **`parent-win` «выигрывает» — но это артефакт ШИРИНЫ ЮНИТА, не лучшее ранжирование.** Окно ±1 кладёт в слот ~3 чанка (3.00 / 8.48 / 13.90 против 1/3/5) → покрытие на слот растёт **механически**. При **равном бюджете контекста** честное сравнение разворачивается наоборот: `hybrid+rerank` top-**3** (3 чанка) Recall **0.935** > `parent-win`@1 (3 чанка) Recall **0.823**. Умный reranker top-k покрывает больше gold на тот же бюджет токенов, чем «слепое» окно индекс-соседей вокруг top-1.

### Почему так (разбор)
- **Section:** гранулярность чанкинга на этом корпусе совпала с гранулярностью секций → «родителя крупнее чанка» физически нет.
- **Window:** окно ±1 берёт **индекс-соседей**, а не лучшие по релевантности чанки. Reranker top-k бьёт окно при равном числе чанков. Block-@k считает **блоки**, а блок ≈ 3 чанка → харнес легко «обмануть», расширив юнит.
- Настоящий вопрос parent-document — лучше ли отдать LLM **непрерывную окрестность** хита (контекст в порядке чтения), чем k **разрозненных** top-чанков на **тот же** бюджет токенов — retrieval-харнес **не видит** (он меряет только покрытие чанков). Это вопрос **RAGAS** (faithfulness / answer_relevancy).

### Честные конфаунды / границы инструмента
- Block-@k **премирует широкий юнит**. Чтобы сравнение было честным, держим бюджет контекста постоянным — репортим chunks-delivered рядом с покрытием (сделано выше).
- `parent-sec` бесполезен именно потому, что чанкер уже «секция-в-чанк»; на корпусе длинных секций вывод мог бы отличаться (граница применимости, не баг реализации).

### Решение по дефолту
- `parent_retrieval="off"` **остаётся**. `parent-sec` — no-op на этом корпусе; `parent-win` раздувает контекст ×3 без выигрыша на токен. Off за флагом (как HyDE/semantic/late). Код сохранён для (а) корпусов с многочанковыми секциями, где `section` реально склеивает, и (б) будущего RAGAS-сравнения «непрерывная окрестность vs disjoint top-k».

> **Главный урок Модуля 11:** пятый эксперимент трека C — и снова харнес выносит честный вердикт, но с двумя **мета-уроками**. (1) Результат, идеально совпавший с прогнозом («Δ≈0»), оказался **багом** (no-op), вскрытым только инспекцией механизма — *проверяй причину, а не метрику; «зелёное и как предсказано» — самый коварный вид неправильного.* (2) Retrieval-харнес можно **«обмануть», расширив юнит выдачи**: per-slot покрытие `parent-win` растёт чисто механически, а per-token честное сравнение отдаёт победу обычному top-k. Payoff parent-document (непрерывный контекст для LLM) лежит **вне** retrieval-харнеса → это и есть заранее названный валидный исход: **«context-фича, её эффект вне retrieval-харнеса → следующий приоритетный шаг — RAGAS (Трек A §1, шаг 2)».** Карта трека C: recursive on; HyDE/semantic/late/parent — off (измеренно бесполезны/хуже/вне харнеса). Дальше измеримость двигает не ретрив, а RAGAS.
