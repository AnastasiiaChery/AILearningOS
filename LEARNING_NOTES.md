## Модуль 1 — Чанкинг и эмбеддинги

### Концепция
- Эмбеддинг-модель превращает текст в **один вектор фикс. длины** (384). Один вектор на весь документ «усредняет» смысл → деталь тонет. Поэтому документ режут на **чанки**, каждый → свой вектор, поиск идёт по чанкам.
- **Жёсткий факт:** у каждой модели есть лимит входа в токенах. У `all-MiniLM-L6-v2` это **256 токенов**. Всё длиннее — **молча обрезается**, остаток текста никогда не попадает в вектор.
- `normalize_embeddings=True` → векторы единичной длины, тогда **cosine == dot product**, что и ожидает Qdrant с `Distance.COSINE`.

### Проблема в коде (было)
- `_estimate_tokens = len(text)//4` — «токены» считались по символам, а не реальным токенайзером.
- `_split_by_size`: окно бралось как `max_tokens*4` **слов** (`512*4=2048` слов), хотя комментарий про «4 символа на слово». Токены/слова/символы перемешаны.
- `chunk_size=512` в конфиге — **вдвое больше лимита модели (256)**.
- Баг **латентный**: на мелких документах секции маленькие, до `_split_by_size` дело не доходит — всё «выглядит нормально».

### Моё рассуждение (разбор на примере секции 1500 слов)
1. В один чанк попадёт **~2048 слов** (по формуле `words[start : start + 512*4]`).
2. Это **~2662 реальных токена** (1 слово ≈ 1.3 токена).
3. → Модель **обрежет** до первых 256 токенов.

**Поправка (ключевой "ага"):** 256 из 2662 — это **не «чуть больше половины», а < 10%**. То есть в вектор попадал примерно первый абзац, а **~90% секции выбрасывалось** и было невидимо для поиска. Чанк при этом лежит в БД «целиком» — поэтому баг и коварен.

### Моя гипотеза «нужно делать чанки меньше» — и важная развилка
Смешались **две разные вещи**:
1. **Потеря из-за обрезки** лечится **не** уменьшением чанка, а тем, что мы режем секцию на **много** чанков и покрываем её **целиком** (+ overlap, чтобы мысль на стыке не рвалась). Информация больше не теряется в принципе.
2. **Размер чанка** — отдельный **рычаг точности**, классический trade-off:

| Чанки меньше (128–180) | Чанки больше (≈256) |
|---|---|
| острее семантика, меньше шума | больше контекста в одном куске |
| идея может разорваться; больше точек в БД | вектор «усредняет» смысл, факт может раствориться |

Вывод: «меньше чанк» **не возвращает** потерянную инфу (её возвращает разбиение), а повышает точность ценой контекста. Правильный инженерный путь — **разумный дефолт (256) + замер** (настроим эмпирически в Модуле 3 с reranker’ом), а не подбор «на глаз».

### Решение (стало)
- Счёт и нарезка — **родным токенайзером модели** (`get_embedder().tokenizer`): лимит 256 определён именно в её WordPiece-токенах, поэтому это единственный честный источник истины (tiktoken от OpenAI — другой словарь, дал бы расхождение).
- `_split_by_tokens`: sliding-window по реальным id с overlap по токенам, decode обратно в текст.
- `_effective_max_tokens = min(chunk_size, model_max) - 16`. **Headroom 16** — на спец-токены `[CLS]/[SEP]` **и** на дрейф round-trip: окно из N id, декодированное в текст и заново закодированное при эмбеддинге, может вернуться как на пару токенов больше (`decode∘encode` у WordPiece **не тождественен**).
- `embedder.py`: добавлены `aembed_texts/aembed_query` через `asyncio.to_thread` — `model.encode` это блокирующая CPU-операция, прямой вызов в async **заморозил бы event loop** на всё время кодирования.

### Проверка на данных
- Секция **1440 слов**: было — 1 обрезанный чанк (теряем ~90%); стало — **9 чанков по ≤240 токенов**, покрытие полное.
- `tests/test_chunker.py` — 4 регрессии (≤ лимит, многокусочное разбиение, headroom, покрытие первого/последнего токена) — **зелёные**.

## Модуль 2 — Sparse / BM25 и гибридный поиск *(в процессе)*

### Концепция
- **Dense вектор** — семантика (384 числа). Ловит перефразирование (*«поиск по смыслу»* ≈ *«semantic retrieval»*). Слепое пятно — **точные редкие термины** (имена функций, артикулы, аббревиатуры вроде `RRF`).
- **Sparse вектор** — лексика. Размер во весь словарь (десятки тысяч), почти всё нули; ненулевые = веса реально встретившихся слов. Ловит **точное совпадение**.
- **BM25** — классические веса sparse:
  - **TF** — чем чаще слово в документе, тем выше вес, но с **насыщением** (5-е упоминание < 2-го).
  - **IDF** — чем в **меньшем числе документов корпуса** слово встречается, тем оно ценнее (`the` ≈ 0; `qdrant` — большой). **IDF считается по всему корпусу, не внутри одного документа.**
- **Гибрид + RRF** (Reciprocal Rank Fusion): объединяем не сырые скоры (косинус и BM25 в разных шкалах), а **ранги**: `score = Σ 1/(k + rank)`. Устойчиво к разным шкалам.

### Проблема в коде (`qdrant_store.py:30-47`, тот же код при загрузке и при запросе)
```python
idx = hash(token) % 65536            # слово → колонка вектора
tf  = count / total
idf = math.log(1 + 1/(count + 1))    # «IDF» по частоте в ОДНОМ тексте
```
Две гипотезы дефекта (проверим экспериментом):
- **А (hash-соль):** `hash()` для строк в Python солится случайно при каждом запуске процесса (`PYTHONHASHSEED`). Если бэкенд перезапущен между загрузкой и запросом — `hash("qdrant") % 65536` даст **другую колонку** → слова перестают совпадать → keyword-поиск тихо деградирует.
- **Б (ложный IDF):** формула использует частоту в одном тексте, а не долю документов корпуса → редким и частым словам даёт почти одинаковый вес → смысл IDF теряется.

### Моё рассуждение (и эксперимент)
- **Часть А** — гипотеза «hash совпадёт» **опровергнута экспериментом**: `hash("qdrant") % 65536` дал 1909 / 38650 / 44106 (хост) и 6896 / 10086 (контейнер) — каждый запуск процесса разный. `PYTHONHASHSEED` не задан → рандомизация включена (дефолт CPython с 3.3). Вывод: после рестарта слово ложится в **другую колонку** → перестаёт находить себя. Тихая деградация (dense ещё что-то находит — кажется, что «работает»).
- **Часть Б** — «формула не даёт одинаковый вес»: **наполовину верно**. Формула не константа (падает с `count`: 0.4055 → 0.0099), но зависит от **неправильной переменной** — частоты **внутри одного текста**. Эксперимент: `the`(1 раз)=0.4055 и `qdrant`(1 раз)=0.4055 — **идентично**; настоящий корпусный BM25-IDF: `the`=0.05, `qdrant`=5.66 (×100). Корень: IDF требует статистики корпуса, а функция видит один текст за раз.

### Решение (свой TF + стабильный хеш + Qdrant IDF) — выбрано осознанно
1. **Стабильный токен→id**: `blake2b(token)` в пространство `2^20` вместо солёного `hash()` → детерминирован между процессами (чинит дефект №1).
2. **В вектор кладём только TF** c BM25-насыщением `count*(k1+1)/(count+k1)`, `k1=1.5`, **без** length-norm (он требует корпусного avgdl) и **без** самодельного idf.
3. **IDF делегирован Qdrant** через `SparseVectorParams(modifier=Modifier.IDF)` в `lifespan.py` — сервер считает IDF по реальной document-frequency корпуса при запросе (чинит дефект №2, переживает рестарты).
4. **Агрегация по индексу** (коллизии хеша) — Qdrant требует уникальные индексы в sparse-векторе.
5. Стоп-слова **не нужны списком** — корпусный IDF сам придавливает их к нулю.
6. ⚠️ Коллекцию пришлось **пересоздать**: модификатор задаётся при создании; existing-коллекция без него не обновится → delete + restart (lifespan создаёт заново) + re-ingest.

### Проверка на данных
- **Детерминизм/рестарт:** `token_id('rrf')=462471`, `token_id('qdrant')=277249` — **идентичны до и после рестарта**; sparse-only поиск «RRF…» → чанк «Hybrid Search», score **7.4133**, тот же после рестарта.
- **IDF-эффект (корпусный):** sparse-score `qdrant`/`hybrid`=1.674 (редкие), `the`=0.297 (стоп-слово придавлено автоматически). Нюанс: IDF относителен к корпусу; на 7 чанках статистика шумная (`is`=1.18, т.к. реже `the`).
- **Полный гибрид:** `hybrid_search` → 5 чанков, dense+sparse→RRF, top = «Hybrid Search» (rrf_score 1.0).
- `tests/test_sparse.py` (детерминизм, уникальные индексы, насыщение TF, «нет idf в значениях») + `test_chunker.py` — **8/8 зелёные**.

### Открытый хвост (не блокер)
- `qdrant-client` в образе — **1.18.0**, сервер — **1.12.4**: предупреждение о несовпадении minor-версий (>1). Сейчас всё работает; чисто было бы выровнять (поднять образ Qdrant до 1.18.x или закрепить клиент ~1.12). Решим при желании.

## Модуль 3 — Reranking

### Концепция
- **Bi-encoder** (наш `all-MiniLM-L6-v2`): кодирует запрос и документ **раздельно** в векторы, сравнивает косинусом. Вектор документа можно **предпосчитать** при загрузке → быстрый поиск. Цена — грубость: в момент кодирования документ «не знает» о запросе, нюанс усреднён.
- **Cross-encoder** (reranker `cross-encoder/ms-marco-MiniLM-L-6-v2`): берёт пару **(запрос, документ) вместе**, cross-attention позволяет токенам запроса смотреть на токены документа → точная релевантность одним числом. Цена — **нечего предпосчитать**, нужен forward-pass на каждую пару при запросе.
- **Двухстадийная схема:** Стадия 1 (гибрид, дёшево) → top-N кандидатов (recall). Стадия 2 (cross-encoder, дорого) → пересчёт N пар, top-k (precision).

### Моё рассуждение
- **А:** cross-encoder видит обе стороны одновременно → анализирует **взаимодействие** токенов (cross-attention), а не расстояние между двумя готовыми векторами. Bi-encoder сжимает каждую сторону до их «встречи» — нюанс уже потерян.
- **Б:** дорого, т.к. на каждый запрос нужен прогон трансформера для **каждой** пары; ничего не предпосчитать. Узкое место при 100k чанков: reranker = **O(N) прогонов/запрос** (секунды-минуты), а bi-encoder+ANN = **~O(1) кодирование + O(log N) поиск** (вектора посчитаны 1 раз при загрузке). Поэтому reranker — только по маленькому пулу кандидатов.

### Решение
- Новый сервис `services/rag/reranker.py`: `CrossEncoder` синглтон (`lru_cache`), `rerank_sync` (scores → sort → top_k), `arerank` через `asyncio.to_thread` (CPU-bound, как эмбеддер).
- `retriever.hybrid_search(query, top_k, rerank)`: стадия 1 тянет `rerank_candidates=20` (шире для recall), стадия 2 — cross-encoder → `retrieval_top_k=5`. Флаг `rerank` позволяет сравнивать с чистым RRF.
- Конфиг: `rerank_enabled`, `rerank_candidates`, `reranker_model`. Reranker греется на старте в `lifespan.py`.
- Зависимостей не добавляли — `CrossEncoder` уже в `sentence-transformers` (модель качается с HF в рантайме, как эмбеддер).

### Проверка на данных
Запрос *«why does a cross-encoder improve the order of search results»* по обогащённому корпусу:
- **RRF only (top-5):** скоры плоские (1.0, 0.58, 0.50, 0.42, 0.33).
- **+ rerank (top-5):** единственный релевантный кусок «Why Reranking Helps» = **+5.82**, остальные **−8…−11** — резкое разделение. Состав top-5 переставлен (вошла Quantization из пула 20, выпала Cosine).
- Вывод: cross-encoder даёт **дискриминативный** сигнал (что реально отвечает на вопрос), RRF — лишь относительный ранг.
- `tests/test_reranker.py` (ранжирование по релевантности + усечение + пустой вход) — часть набора **10/10 зелёных**.

## Модуль 4 — Дизайн агентов LangGraph

### Концепция
- **Цепочка/фиксированный граф** — путь зашит. Наш mentor: `retrieve → generate`, всегда ищет один раз сырым последним сообщением.
- **Агент** — LLM сам решает (через tool calls): искать или нет, какой запрос, сходить ли ещё раз (multi-hop), когда остановиться.

### Проблема (показано на данных)
Follow-up-запросы достают мусор, т.к. retrieve берёт сырой `user_message`:
- `What is RRF…` → топ ce=+8.65 (ок); `can you give an example of that?` → ce=−3.19; `tell me more about it` → ce=−9.34 (всё нерелевантно — слова `it/that` без темы). Контекст диалога терялся на этапе поиска.

### Моё рассуждение
1. Нужную тему хранит **история диалога** (`state["history"]`).
2. Превращение «that» → «Reciprocal Rank Fusion» — работа **LLM**, не кода; шаг встаёт **новым узлом перед retrieve**.

### Решения (3 фикса)
**№6 — контекстуализация запроса.** Новый узел `contextualize_query`: `history + user_message → standalone search query` через LLM (только при наличии истории — первое сообщение самодостаточно, экономим вызов). Граф: `contextualize → retrieve → generate`. Поле `search_query` в `MentorState`. Проверка: `can you give an example of that?` → **«example of Reciprocal Rank Fusion in hybrid search»**; `tell me more about it` → **«What is Reciprocal Rank Fusion in hybrid search»**.

**№7 — настоящий стриминг.** Было: `ainvoke` целиком → разбивка по словам (пользователь ждёт весь ответ). Стало: узел `generate` использует `llm.astream` (`streaming=True`), а `chat.py` читает граф через `astream(stream_mode=["messages","values"])` и форвардит токены **только узла `generate`** (узел `contextualize` тоже зовёт LLM — его поток фильтруем). Цитаты берём из финального снапшота `values`. Проверка: **56 токен-чанков по одному** (`In`→` hybrid`→` search`→`,`→`R`→`RF`), а не 1 блоб.
> Урок про неудачную попытку: первый заход (`stream_mode="messages"` + узел на `ainvoke`, `streaming=False`) дал **1 чанк** — LangGraph стримит токены, только если **сам узел стримит модель**. Поэтому перешли на `llm.astream` внутри узла.

**№8 — structured output в планнере.** Было: ручное срезание ```-заборов + `json.loads` + петля ретраев. Стало: `get_llm().with_structured_output(PlanModel)` — модель принуждается вернуть валидный объект (Pydantic-схема `PlanModel/TopicModel/SubtopicModel`), парсинг и ретраи удалены, граф линеен (`generate → persist`), есть fallback на минимальный план. Проверка: план генерится (HTTP 201, 5 топиков с часами/субтопиками).

### Проверка на данных
- Контекстуализация, стриминг (56 чанков), structured-план (201) — все зелёные.
- Полный набор тестов (chunker 4 + sparse 4 + reranker 2) — **10/10**.

### Вершина (опционально, не делали) — настоящий tool-calling агент
Сейчас mentor всё ещё фиксированный граф: **всегда** ищет ровно один раз. Можно дать LLM `retrieve` как **инструмент** и позволить ей самой решать: нужен ли поиск (на «привет» — нет), какой запрос послать, сходить ли в базу несколько раз (multi-hop для составных вопросов), когда хватит контекста. В LangGraph это паттерн `create_react_agent` / цикл `agent ⇄ tools` с условным ребром. Это следующий уровень — обсудим/сделаем при желании.
